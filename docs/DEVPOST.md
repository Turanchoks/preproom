## Headline / Tagline

**TutorRoom — an autonomous prep room for tutors: lesson memory, homework, media, and video analysis for every student.**

---

## Inspiration

Independent language tutors spend roughly 45 minutes preparing for each student before every lesson — reviewing what happened last time, writing the next plan, drafting homework by hand. Multiply that by 15 students and it is an entire workday every week, unpaid. We asked: what if a tutor could upload last night's lesson recording and come back to find the next plan and homework already drafted — targeted at exactly where the student struggled, with audio clips and image flashcards included?

The answer is not a smarter chatbot. It is a **persistent agent per student** that accumulates evidence across sessions and acts on it autonomously. That is what TutorRoom is.

---

## What it does

TutorRoom is a per-student AI studio for language tutors. Every student gets their own ADK agent with long-term agentic memory. The teacher opens a student, chats with that student's agent, and watches lesson plans and interactive homework stream onto a live canvas next to the chat.

**Key capabilities:**

- **Agentic memory.** The agent decides what is worth remembering (`save_fact` tool) — recurring errors, level milestones, interests, lesson coverage — and surfaces it in every subsequent chat. This is not transcript-stuffing; the agent is the author of its own memory.
- **Interactive homework.** Homework is generated as structured JSON (Gemini structured output, validated against Zod schemas) and renders as a live quiz: multiple choice, fill-in-the-blank, word matching, gap fill, word puzzles, sentence matching, **listening exercises** (audio generated by Gemini TTS) and **image flashcards** (illustrations generated by Gemini image generation). A pedagogy-critic sub-agent reviews every generated set against an embedded rubric before it is surfaced to the teacher.
- **Video pipeline.** The teacher uploads a lesson recording. It goes to Cloud Storage, a Pub/Sub message triggers an OIDC-verified push endpoint, and Gemini multimodal watches the actual video against a teaching rubric — extracting what was covered, where the student struggled, what they mastered. The analysis lands as a report artifact and as memory facts in the student's permanent record.
- **Autonomous proactive loop.** As soon as video analysis completes, a second ADK `LlmAgent` runs **with no human turn** (the SalesShortcut pattern): it reads the analysis, drafts the next-lesson plan and a homework set targeting the student's specific struggles, and records a "Proactive prep ready" memory note — all persisted before the teacher even opens the app. The agent tells the teacher about it in the next chat unprompted.
- **Share links + closed learning loop.** Any artifact can be shared with one click. Students open `/s/[slug]` — no account — and complete the homework with instant feedback and TTS audio. When they finish, results (score, per-exercise pass/fail) are posted back to the server as memory facts: `"Homework 'Past Simple Drill': scored 4/5"` plus one error fact per failed exercise. The next time the teacher asks "How did Anna do?", the agent answers from authoritative evidence.
- **MCP exercise catalog.** A real Model Context Protocol stdio server exposes the exercise type catalog to the agent. The agent queries it when choosing exercise types for a homework set.

## How we built it

**Agent layer — Google ADK (TypeScript)**

The core is an ADK `LlmAgent` backed by `gemini-3.5-flash`. Each request builds a fresh agent whose system instruction is compiled from the student's profile and their current memory facts (retrieved from Postgres at request time). The agent has 12 function tools: `save_fact`, `search_memory`, `create_lesson_plan`, `create_homework`, `update_artifact`, `get_student_profile`, `list_videos`, `get_video_analysis`, `generate_illustration`, `generate_audio_snippet`, `web_search` (an AgentTool sub-agent), and `pedagogy_reviewer` (a second LlmAgent exposed as an AgentTool that reviews homework against a pedagogy rubric). Every tool closes over a single `studentId` — cross-student access is structurally impossible.

The proactive loop (`lib/agent/proactive.ts`) reuses the same tool set with a no-op stream writer; artifacts still land in the DB because the persistence calls are inside the tool implementations, not in the stream consumer.

**ADK ↔ AI SDK bridge**

The chat UI is the Vercel AI SDK streaming protocol (UI message parts). ADK emits its own event stream (text deltas, function calls, function responses). We wrote a bridge (`app/(chat)/api/chat/route.ts`) that converts ADK events into AI SDK data parts, including the artifact-canvas choreography (kind → id → title → clear → content deltas → finish). This is the most brittle piece of the system — we spent real time on the streaming edge cases (partial JSON, concurrent events, early termination).

**Homework generation**

Gemini structured output with `responseJsonSchema` derived from the Zod schema per exercise type (`lib/quiz/homework-schema.ts`). The model generates JSON that is immediately parsed and validated; invalid exercises are dropped, never forwarded. After the JSON is finalized, a media post-processor (`lib/media/`) generates WAV files (Gemini TTS) and PNG illustrations (Gemini image generation) for exercises that need them, caches them by content hash, uploads to Cloud Storage, and injects the URLs before the artifact is saved.

**MCP server**

A real stdio MCP server (`mcp/`) exposes the exercise catalog as queryable tools. In production it is spawned as a child process; in development there is an in-process `FunctionTool` fallback so the app runs without the subprocess.

**Video pipeline**

Upload → `@google-cloud/storage` → Pub/Sub publish → OIDC-verified push webhook → `lib/analysis/video-analyze.ts` calls Gemini multimodal with the video file URI and a structured analysis prompt (rubric in `docs/harvest/pedagogy-rubric.md`) → structured output is parsed and written as a `Video` record + memory facts. Locally `PUBSUB_MODE=direct` runs the analysis in-process without any GCP resources.

**Infrastructure**

Single Cloud Run service (Next.js 16 + embedded ADK runtime). Cloud SQL Postgres 16 via unix socket (`/cloudsql/...`). Cloud Storage for videos and generated media. Pub/Sub with an OIDC-authenticated push subscription for the video pipeline. Deployment is a single `./infra/deploy.sh` script that wires all of this in one call.

**Frontend**

Next.js 16 (App Router, React 19), Vercel AI SDK v6 UI, Tailwind v4 + shadcn/ui. Base was the `vercel/ai-chatbot` template; the student sidebar, per-student pages, studio layout, quiz player, share pages, and landing page were all built on top of it.

## Challenges we ran into

**1. ADK ↔ AI SDK streaming bridge.** The Vercel AI SDK UI protocol expects a specific sequence of typed data parts for artifact canvas control (kind → id → title → clear → deltas → finish). ADK's event stream is not ordered the same way and can interleave text deltas with function-call events. The bridge has to buffer, reorder, and translate without losing tokens or misidentifying which events belong to which artifact. We had streaming artifacts that would corrupt or close prematurely until we got the sequencing right.

**2. Free-tier quota wall.** The Gemini free tier is 20 requests per day. We hit it about two hours into development. The fix was simple — provision a billed GCP project and mint a new key — but it cost us time and temporarily blocked image generation and TTS. After that we moved everything to the billed project (`teachflow-hack-611`).

**3. Postgres unix sockets on Cloud Run.** Cloud SQL from Cloud Run uses a unix socket (`/cloudsql/<connection-name>`). The `postgres-js` client (used by Drizzle) needs the connection string formatted as `postgresql://user:pass@localhost/db?host=/cloudsql/...`. Getting this right across local dev (TCP) and Cloud Run (unix socket) without two separate connection strings required an environment-flag branch in the DB setup. Two deploy attempts failed before we got it.

**4. Homework JSON partial-stream parsing.** Gemini structured output streams the JSON token by token. The homework delta handler (`artifacts/homework/server.ts`) accumulates the full JSON, parses it once at the end, and then sends it to the canvas — but during streaming the canvas needs to show *something*. We send the raw JSON chunks as `homeworkDelta` parts and do the final parse-and-validate only at `data-finish`. This means the canvas briefly shows a loading state rather than partial exercises, which is actually better UX than a partially valid exercise list.

**5. Media generation after structured output.** Gemini TTS and image generation are separate API calls that must happen after the homework JSON is finalized (because the exercise JSON contains the `prompt` and `imagePrompt` fields). The post-processor runs these in parallel (concurrency 3) but they still add 5–10 seconds to total generation time. We made them non-blocking from the user perspective: the homework artifact saves and the canvas shows it; the media URLs are injected by a background update.

## Accomplishments we are proud of

- A genuine autonomous loop: the agent drafts artifacts with no human turn, triggered by video analysis completion, and the teacher discovers them in the next session.
- The closed learning loop: student quiz results become memory facts, so the agent has ground-truth performance data, not just teacher impressions.
- Homework that passes 100% schema validation and scores 4.47/5 on an LLM-judged pedagogy rubric across 10 golden briefs (CEFR A1–C1, 8 languages).
- A working MCP stdio server wired into an ADK agent — not hardcoded, a real MCP integration with a fallback.
- The ADK ↔ AI SDK bridge: token-level streaming UX on top of an ADK agent, with a live artifact canvas.
- Zero GCP resources needed for local development: same code, env flags switch modes.

## What we learned

- The ADK `AgentTool` pattern (one LlmAgent exposed as a tool to another) is genuinely useful for critic loops and sub-agents — it is not just a demo trick. The pedagogy reviewer runs as a real second agent and catches generation issues.
- Gemini structured output with `responseJsonSchema` is reliable enough to use as a hard contract for UI-rendered content — but you still need a fallback parse + filter for exercises that slip through malformed.
- Streaming + structured output is an unsolved UX problem. The user wants to see progress; the validator wants a complete document. We settled on buffering the full JSON for validation and streaming progress indicators separately.
- Cloud Run + Cloud SQL unix sockets is the right deployment pattern for a hackathon; the complexity is a one-time setup cost and after that deploys are fast.
- An autonomous background agent that produces artifacts the teacher finds waiting for them is the single most compelling demo beat. Build that first.

## What's next

- **Parent progress reports** — auto-generated summaries of student progress, shareable with parents, drawn from the memory fact store.
- **Homework results analytics** — dashboards over the results that already flow into agent memory: per-student error trends, exercise-type performance, engagement rates.
- **Multi-teacher studios** — shared student rosters across a small teaching studio, role-based access, studio-level billing.
- **Agent Engine migration** — move the proactive agent loop to Google Cloud Agent Engine for managed state, retries, and scheduling, replacing the current in-process trigger.
- **A2A task endpoint** — the agent card at `/.well-known/agent.json` is already published; a full A2A task endpoint would let other agents (e.g., a scheduling agent) commission TutorRoom to prepare materials for a specific student.
- **Google Cloud Marketplace listing** — distribution to GCP-billed organizations (Track 3 trajectory).
